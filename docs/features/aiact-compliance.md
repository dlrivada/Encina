## Overview

Encina.Compliance.AIAct provides attribute-based EU AI Act compliance enforcement at the CQRS pipeline level:

| Component | Description |
|-----------|-------------|
| **`[HighRiskAI]` Attribute** | Marks request types as high-risk AI operations with category and system ID (Art. 6) |
| **`[RequireHumanOversight]` Attribute** | Requires human review before request execution (Art. 14) |
| **`[AITransparency]` Attribute** | Declares transparency obligations with disclosure text (Art. 13/50) |
| **`AIActCompliancePipelineBehavior`** | Pipeline behavior that evaluates all AI Act attributes and compliance checks |
| **`IAIActClassifier`** | Risk classification, prohibited practice detection, and compliance evaluation |
| **`IAISystemRegistry`** | Central registry for AI system registrations with reclassification history |
| **`IAIActComplianceValidator`** | Generic validation across request types with attribute-based system ID resolution |
| **`IHumanOversightEnforcer`** | Human decision recording and approval tracking |
| **`IDataQualityValidator`** | Training data quality assessment with bias detection (Art. 10) |
| **`IAIActDocumentation`** | Technical documentation generation and updates (Art. 11) |
| **`AIActOptions`** | Configuration for enforcement mode, auto-registration, health check |

### Why Pipeline-Level AI Act Compliance?

| Benefit | Description |
|---------|-------------|
| **Automatic enforcement** | Compliance is validated whenever a decorated request enters the pipeline |
| **Declarative** | AI Act requirements live with the request types, not scattered across services |
| **Transport-agnostic** | Same compliance enforcement for HTTP, message queue, gRPC, and serverless |
| **Non-negotiable safety** | Prohibited practices (Art. 5) are always blocked regardless of enforcement mode |
| **Pluggable** | Replace any default service (registry, classifier, oversight) with custom implementations |
| **Observable** | Every check emits OpenTelemetry traces, metrics, and structured log events |

## The Problem

The EU AI Act (EU 2024/1689) mandates that AI systems be classified by risk level, with specific obligations for each tier. High-risk systems require human oversight, transparency disclosures, technical documentation, and data quality guarantees. Prohibited practices must never be deployed.

Without a framework-level solution, compliance logic scatters across controllers, services, and middleware — creating gaps, inconsistencies, and audit risks.

## The Solution

Encina.Compliance.AIAct centralizes AI Act compliance at the CQRS pipeline level:

1. **Decorate** request types with `[HighRiskAI]`, `[RequireHumanOversight]`, `[AITransparency]`
2. **Configure** enforcement mode (Block/Warn/Disabled) and auto-registration
3. **The pipeline** validates compliance before request execution
4. **Non-compliant requests** return `EncinaError` (Block mode) or log warnings (Warn mode)
5. **Prohibited practices** are always blocked, regardless of enforcement mode

## Quick Start

```csharp
// 1. Register services in Program.cs / Startup.cs
services.AddEncinaAIAct(options =>
{
    options.EnforcementMode = AIActEnforcementMode.Block;
    options.AutoRegisterFromAttributes = true;
    options.AssembliesToScan = [typeof(Program).Assembly];
});

// 2. Decorate your request types
[HighRiskAI(Category = AISystemCategory.EmploymentWorkersManagement, SystemId = "cv-screener")]
[RequireHumanOversight(Reason = "Employment decisions require human review")]
[AITransparency("This screening result was generated by AI.")]
public sealed record ScreenCandidateCommand(string CandidateId) : ICommand<ScreeningResult>;

// 3. Send the command — compliance is enforced automatically
var result = await encina.Send(new ScreenCandidateCommand("C-001"));
result.Match(
    Right: screening => /* compliant, proceed */,
    Left: error => /* non-compliant, error.Code starts with "aiact." */
);
```

## HighRiskAI Attribute

Marks a request type as a high-risk AI operation per Art. 6 and Annex III.

```csharp
// Basic usage with category
[HighRiskAI(Category = AISystemCategory.CriticalInfrastructure)]
public sealed record MonitorPowerGridCommand(string GridId) : ICommand<GridStatus>;

// With explicit system ID (overrides auto-generated ID based on type name)
[HighRiskAI(Category = AISystemCategory.BiometricIdentification, SystemId = "face-match-v2")]
public sealed record IdentifyPersonCommand(byte[] FaceData) : ICommand<IdentityMatch>;
```

## RequireHumanOversight Attribute

Requires human review before the request is executed (Art. 14).

```csharp
[RequireHumanOversight(Reason = "Loan decisions above €50k require human approval")]
public sealed record ApproveLoanCommand(string ApplicationId, decimal Amount) : ICommand<LoanDecision>;
```

Human decisions are recorded via `IHumanOversightEnforcer`:

```csharp
var enforcer = serviceProvider.GetRequiredService<IHumanOversightEnforcer>();

// Record a human decision
await enforcer.RecordHumanDecisionAsync(new HumanDecisionRecord
{
    DecisionId = Guid.NewGuid(),
    SystemId = "loan-approver",
    ReviewerId = "reviewer-42",
    ReviewedAtUtc = DateTimeOffset.UtcNow,
    Decision = "approved",
    Rationale = "Risk assessment within acceptable parameters"
});

// Check if approval exists
var result = await enforcer.HasHumanApprovalAsync(decisionId);
```

## AITransparency Attribute

Declares transparency obligations per Art. 13 and Art. 50.

```csharp
// AI-generated content disclosure
[AITransparency("This content was generated by AI.")]
public sealed record GenerateReportCommand(string Topic) : ICommand<Report>;

// Deepfake disclosure
[AITransparency("This image has been synthetically generated.",
    ObligationType = TransparencyObligationType.Deepfakes)]
public sealed record GenerateImageCommand(string Prompt) : ICommand<Image>;

// Chatbot interaction disclosure
[AITransparency("You are interacting with an AI assistant.",
    ObligationType = TransparencyObligationType.ChatbotInteraction)]
public sealed record ChatCommand(string Message) : ICommand<ChatResponse>;
```

## Prohibited Practices

Eight prohibited AI practices from Art. 5 are **always blocked**, regardless of enforcement mode:

| Practice | Description |
|----------|-------------|
| `SocialScoring` | Social scoring by public authorities |
| `RealTimeBiometric` | Real-time remote biometric identification in public spaces |
| `EmotionRecognition` | Emotion recognition in workplace/education |
| `PredictivePolicing` | Individual predictive policing |
| `UntargetedFacialImageScraping` | Untargeted scraping of facial images |
| `SubconsciousManipulation` | Subliminal/manipulative techniques |
| `VulnerabilityExploitation` | Exploitation of vulnerable groups |
| `BiometricCategorisationByRace` | Biometric categorization inferring race, religion, etc. |

## Enforcement Modes

| Mode | Non-Compliant Behavior | Prohibited Practices |
|------|----------------------|---------------------|
| **Block** | Returns `EncinaError` (request rejected) | Always blocked |
| **Warn** | Logs warning, request proceeds | Always blocked |
| **Disabled** | No compliance checks (passthrough) | Always blocked |

## Error Handling

All errors follow the Railway Oriented Programming pattern with `Either<EncinaError, T>`:

```csharp
var result = await encina.Send(new ScreenCandidateCommand("C-001"));
result.Match(
    Right: screening => HandleSuccess(screening),
    Left: error => error.Code switch
    {
        "aiact.prohibited_use" => HandleProhibitedUse(error),
        "aiact.compliance_validation_failed" => HandleViolations(error),
        "aiact.human_oversight_required" => HandleOversightRequired(error),
        _ => HandleGenericError(error)
    }
);
```

## Best Practices

1. **Start with Warn mode** during development, switch to Block for production
2. **Always use explicit SystemId** on `[HighRiskAI]` for stable registry keys
3. **Combine attributes** — a single request can have `[HighRiskAI]` + `[RequireHumanOversight]` + `[AITransparency]`
4. **Never bypass Art. 5** — prohibited practices cannot and should not be overridden
5. **Register custom implementations** before calling `AddEncinaAIAct()` to override defaults (TryAdd semantics)
6. **Use auto-registration** for consistent registry population across deployments

## FAQ

**Q: Can I disable prohibited practice blocking?**
A: No. Art. 5 prohibited practices are always blocked regardless of enforcement mode. This is by design — the EU AI Act makes these non-negotiable.

**Q: What happens if a request has no AI Act attributes?**
A: It bypasses all compliance checks with zero overhead. The pipeline behavior short-circuits immediately.

**Q: Can I use this without the pipeline behavior?**
A: Yes. The `IAIActClassifier`, `IAISystemRegistry`, and `IHumanOversightEnforcer` interfaces can be used directly for programmatic compliance checks.

**Q: Does this require a database?**
A: No. The default `InMemoryAISystemRegistry` uses `ConcurrentDictionary`. For production, implement `IAISystemRegistry` with your preferred persistence layer.
